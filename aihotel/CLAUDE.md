## Что это за проект

AI-система управления коттеджами и хостелами ВК "Губаха" (горнолыжный курорт, Пермский край). Два Python-скрипта-агента автоматизируют еженедельный цикл: анализ финансов → постановка задач в Google Таблице. Коммуникация с командой — только через таблицу, без мессенджеров. Подробно: [project.md](project.md).

Разработка ведётся вайбкодингом. Разработчик — Python-новичок. Код должен быть максимально простым и читаемым.

## Команда (6 исполнителей)

Агент 2 ставит задачи именно этим людям. Зоны ответственности критичны для правильных задач.

| Имя | Роль | Зона ответственности | Местоположение |
|-----|------|----------------------|----------------|
| Виктор | Руководитель проекта | Операционное управление объектом, координация команды, внесение данных в таблицу | Удалённо |
| Евгения | Руководитель отдела бронирования и групповых продаж | TravelLine, Bitrix24, OTA (Booking, Островок и др.), администраторы ресепшена, групповые заезды | Удалённо |
| Надежда | Маркетинг | Акции, СММ, дизайн, лидогенерация, пакеты ДР, привлечение групп | Удалённо |
| Управляющий | Управляющий объектом | Контроль выполнения задач ЦО на объекте, дисциплина персонала, гостевой сервис | На объекте |
| Технический директор | Главный инженер (временно ведёт 2 роли) | Техническая исправность комплекса, ремонт, технический персонал + функции менеджера НФ | На объекте |
| Менеджер НФ | Менеджер номерного фонда (вакансия → сейчас Тех. дир.) | Горничные, графики выходов, качество уборок, расходование ТМЦ | На объекте |

> Управляющий — временная позиция, скоро будет замена. Агент 2 ставит задачи действующему человеку на этой позиции.
> Технический директор ведёт функции двух ролей. Агент 2 учитывает двойную нагрузку и не перегружает.

## Переменные окружения

Хранятся в `.env` (локально) и GitHub Secrets (в Actions).

```
GROQ_API_KEY=                  # Ключ Groq API (console.groq.com)
GOOGLE_CREDS_JSON=             # СОДЕРЖИМОЕ credentials.json (весь JSON одной строкой)
FINANCE_SHEET_ID=              # ID финансовой Google Таблицы (из URL таблицы)
STRATEGY_SHEET_ID=             # ID стратегической Google Таблицы (из URL таблицы)
TL_CLIENT_ID=                  # TravelLine API client id
TL_CLIENT_SECRET=              # TravelLine API client secret
TL_PROPERTY_ID=                # TravelLine ID объекта
```

> **Важно про GOOGLE_CREDS_JSON:** это не путь к файлу, а само содержимое JSON. Скопировать всё содержимое credentials.json и вставить как одну строку. В коде используется `json.loads()`, не `from_service_account_file()`.

> **Про FINANCE_SHEET_ID:** в будущем данные придут автоматически из TravelLine в новую таблицу. FINANCE_SHEET_ID хранится в .env, чтобы при смене таблицы менять только одну переменную.

## Паттерны кода

### Чтение Google Sheets

```python
import os
import json
import gspread
from google.oauth2.service_account import Credentials

def get_client() -> gspread.Client:
    creds_info = json.loads(os.environ['GOOGLE_CREDS_JSON'])
    creds = Credentials.from_service_account_info(
        creds_info,
        scopes=['https://www.googleapis.com/auth/spreadsheets']
    )
    return gspread.Client(auth=creds)

def get_worksheet(client: gspread.Client, sheet_id: str, worksheet_name: str) -> gspread.Worksheet:
    return client.open_by_key(sheet_id).worksheet(worksheet_name)
```

> Создавать `get_client()` один раз в начале скрипта, передавать в функции — не вызывать при каждом чтении ячейки.

### Вызов LLM через Groq (с retry)

```python
import os
import time
from groq import Groq

def call_claude(system_prompt: str, user_message: str) -> str:
    client = Groq(api_key=os.environ['GROQ_API_KEY'])
    for attempt in range(3):
        try:
            response = client.chat.completions.create(
                model='llama-3.3-70b-versatile',
                max_tokens=4096,
                messages=[
                    {'role': 'system', 'content': system_prompt},
                    {'role': 'user',   'content': user_message},
                ],
            )
            return response.choices[0].message.content
        except Exception as e:
            if attempt < 2:
                time.sleep(5)
            else:
                raise RuntimeError(f'Groq API недоступен: {e}')
```

### Работа со статусом системы

Лист «Статус системы» в финансовой таблице — ключи в колонке A, значения в колонке B.

```python
def get_flag(ws_status, key: str) -> str:
    records = ws_status.get_all_records()
    for row in records:
        if row['ключ'] == key:
            return row['значение']
    return 'нет'

def set_flag(ws_status, key: str, value: str) -> None:
    records = ws_status.get_all_values()
    for i, row in enumerate(records):
        if row[0] == key:
            ws_status.update_cell(i + 1, 2, value)
            return

def reset_weekly_flags(ws_status, current_week: str) -> None:
    saved_week = get_flag(ws_status, 'неделя')
    if saved_week != current_week:
        set_flag(ws_status, 'неделя', current_week)
        for key in ['данные_внесены', 'анализ_готов', 'дайджест_записан', 'задачи_сформированы']:
            set_flag(ws_status, key, 'нет')
```

> `reset_weekly_flags` вызывается в начале Агента 1. Если неделя сменилась — все флаги обнуляются автоматически.

## GitHub Actions — расписание

Все времена в UTC. Московское время = UTC+3.

```yaml
# .github/workflows/agents.yml
on:
  schedule:
    - cron: '0 20 * * 1'   # Пн 20:00 UTC = Пн 23:00 МСК → TravelLine Collector
    - cron: '0 23 * * 1'   # Пн 23:00 UTC = Вт 02:00 МСК → Агент 1 (анализ)
    - cron: '0 2 * * 2'    # Вт 02:00 UTC = Вт 05:00 МСК → Агент 2 (задачи)
```

Результат каждого запуска — в Google Таблице. Команда заходит в таблицу сама.

> GitHub Actions cron может опаздывать до 10–15 минут. Для нашего цикла не критично.

## Структура дайджеста Агента 1 (что входит)

Дайджест — это текст, который Агент 2 получает в промпт. Должен содержать:

1. **Ключевые метрики недели** с % отклонением от прошлого года
2. **Красные и жёлтые показатели** — только проблемные (зелёные не перечислять)
3. **Прогресс по стратегическим KPI** — загрузка, выручка, рентабельность (как идём к цели)
4. **Невыполненные задачи прошлой недели** — имя + задача + причина из комментария
5. **Повторно невыполненные** — задачи, которые не выполнены 2 недели подряд (флаг для собственника)
6. **Вывод** — 1-2 предложения: что главный риск этой недели

## Правила кода

- **Простота** — никаких фреймворков оркестрации (LangChain, CrewAI, AutoGen)
- **Состояние в таблице** — никакой локальной БД, Redis, файлов состояния
- **Retry-логика** — реализована в `call_claude()`, не в вызывающем коде
- **Идемпотентность** — каждый агент проверяет флаги перед работой и не делает одно и то же дважды
- **Язык** — комментарии в коде на русском, имена переменных и функций на английском
- **Один скрипт = один агент** — никаких общих импортов между агентами кроме utils/

## Чего не делать

## Как работаем

### Оспаривай мои решения
Если мой подход неоптимальный, если есть лучшее решение — скажи прямо.
Не соглашайся просто потому, что я так сказал. Ты не подхалим — ты советник.
Пример: я прошу «сделай через таблицу» — если для задачи лучше база данных, скажи это и объясни почему.

### Структура ответа
Каждый ответ должен быть в формате:
— Что сделано (конкретно, по фактам)
— Что нужно от меня (решения, уточнения, доступы)
— Следующий шаг (что делаем дальше)

Если всё ясно и следующий шаг очевиден — сразу делай. Не переспрашивай лишний раз.

### Предлагай улучшения
После каждой выполненной задачи — одним предложением скажи что ещё можно автоматизировать или оптимизировать.
Не обязательно реализовывать, но замечай возможности: рутина которая повторяется, потенциал для скилла, дубли в коде.

### Возьми на себя все задачи
Твоя ключевая задача — освободить моё время.
Всё что можешь сделать ты — делай ты. Не спрашивай меня про очевидное.
Если решение есть у тебя — принимай его сам и отчитайся. Не перекидывай мне выбор, который ты можешь сделать лучше.
