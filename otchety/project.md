# Проект: Автоматический еженедельный отчёт — Курорт Губаха

## 1. Контекст и цель

**Объект:** Всесезонный курорт Губаха — единый объект, включающий:
- 33 коттеджа (~180 койко-мест)
- Хостел **Ален** — 14 номеров, 64 койко-места
- Хостел **Даниэль** — 21 номер, 102 койко-места
- Ресторан **Монблан** — 90 посадочных мест

**Проблема, которую решает проект:**
- Еженедельные отчёты заполняются вручную — это занимает время и создаёт риск ошибок и опечаток
- Нет централизованного взгляда на операционную деятельность отеля и ресторана в одном месте
- Нет автоматического выявления проблемных точек и рекомендаций по их устранению

**Цель:** Создать автоматизированную систему, которая каждый понедельник утром самостоятельно собирает данные за прошедшую неделю из Travelline и IIKO, формирует отчёты в Google Таблице и генерирует ИИ-анализ отклонений с рекомендациями.

## 2. Для кого

| Пользователь | Роль |
|---|---|
| Составитель отчёта | Аналитик / операционный менеджер |
| Руководитель проекта | Управляет отелем и рестораном |

Разграничение доступа не требуется — оба пользователя видят всё.

## 3. Источники данных

### Автоматические (через API)

| Система | Данные |
|---|---|
| **Travelline API** | Бронирования, загрузка, выручка по сегментам (коттеджи / Ален / Даниэль), отмены, no-show, каналы продаж, ADR, LOS, повторные гости |
| **IIKO API** | Выручка ресторана (кухня / бар), гости, чеки, время суток, food cost, маржинальность, списания (со/без списания), топ-блюда, оборачиваемость столов |

### Ручной ввод (вносится менеджером раз в неделю)

Часть данных не доступна через API и заполняется вручную в отдельный лист Google Таблицы:
- Количество конфликтных ситуаций с гостями и их решение
- Поломки оборудования и технические неполадки
- Отзывы из внешних источников (Яндекс, 2ГИС, книга отзывов)
- Количество входящих звонков и доля неотвеченных
- Количество уборок (коттеджи / хостелы), в т.ч. стыковочных
- Количество заявок на ремонт и выполненных
- Количество персонала в смену, трансферы
- Прочее (погода, события на курорте, влияющие на спрос)

Исторические данные в обеих системах доступны — используются для сравнения и обнаружения аномалий.

Таблица состоит из **7 листов:**

---

| Отмены | — | Количество и % от бронирований |

| Средняя длина пребывания | LOS (Срок пребывания) | Среднее кол-во ночей |

### Лист 2 — Отель: Сравнение WoW / YoY

Для каждого показателя Листа 1 — три колонки:

| Колонка | Описание |
|---|---|
| Текущая неделя | Фактические данные |
| Аналогичная неделя прошлого года (YoY) | Год к году |
| Отклонение | Абсолютное и % |

Цветовое выделение: рост — зелёный, снижение — красный, в пределах нормы — серый.
Порог сигнала: отклонение >10% — предупреждение; >20% — критический сигнал.

### Лист 3 — Ресторан Монблан: Финансы (текущая неделя + сравнение)

| Показатель | Описание |
|---|---|
| Выручка за неделю | Общая; раздельно: Кухня / Бар (сумма и %) |
| Выручка по времени суток | Утро 9–11 / День 11–17 / Вечер 17–21 |
| Количество гостей | Всего за неделю, среднее в день, по времени суток |
| Средний чек на гостя | Общий; по кухне; по бару |
| Средний счёт (на стол) | — |
| Количество чеков | Всего; по времени суток |
| Оборачиваемость столов | Чеков / кол-во столов; оборот посадочных мест |
| Food cost (себестоимость) | Сумма и % от выручки; раздельно кухня/бар |
| Маржинальная прибыль | Сумма и % (кухня / бар / итого) |
| Списания "без списания" | Количество и сумма аннулированных позиций |
| Списания "со списанием" | Количество и сумма фактических списаний |
| Топ-10 блюд | По выручке и по количеству продаж |
| Антирейтинг блюд | Позиции с падением продаж >20% к прошлому году |
| Выручка от банкетов/мероприятий | Кол-во, сумма, средний чек на гостя/мероприятие |
| Количество отзывов | Яндекс / 2ГИС / книга отзывов; доля негативных |

Каждый показатель — с колонками: текущая неделя / аналогичная неделя прошлого года / отклонение (абсолютное и %).

### Лист 4 — Операционный журнал (ручной ввод)

Еженедельно заполняется менеджером. Структура:

| Блок | Поля |
|---|---|
| Инциденты с гостями | Описание, как решили |
| Технические неполадки | Оборудование, водоснабжение, электричество; статус решения |
| Ремонтные заявки | Кол-во подано / выполнено / не выполнено |
| Уборки | Коттеджи (всего / стыковочных); Хостелы (всего / стыковочных) |
| Звонки | Кол-во входящих; кол-во неотвеченных; % без ответа |
| Отзывы (внешние) | Яндекс / 2ГИС / книга отзывов: количество, оценки, краткое содержание |
| Прочее | Трансферы, погода, события на курорте |

---

### Лист 5 — Сводный дашборд

Один экран с ключевыми показателями по отелю и ресторану за неделю.

**Блок Отель:**
- Загрузка по сегментам (коттеджи / Ален / Даниэль) — светофор
- ADR (Средняя цена), RevPAR (Доход на номер) — vs прошлый год
- Выручка НФ — факт vs прошлый год + спарклайн 8 недель
- Отмены — % и динамика
- Опережающий показатель: забронировано на следующий месяц, % от плана

**Блок Ресторан:**
- Выручка — факт vs прошлый год + спарклайн 8 недель
- Food cost (%) — светофор (норма: исторический коридор)
- Средний чек — динамика
- Гостей за неделю — vs прошлый год
- Списания — сумма "со списанием"

**Блок Операции:**
- Невыполненные заявки на ремонт
- % неотвеченных звонков
- Открытые технические неполадки

Назначение: взгляд на состояние объекта за 30 секунд.

---

### Лист 6 — ИИ-анализ: Проблемные точки и рекомендации

Автоматически генерируется через Claude API (приоритет) или OpenAI GPT API.

**Структура листа:**

#### Раздел А. Выявленные отклонения
Список аномалий, определённых системой на основе сравнения с историческими данными. Пример:

> — Загрузка хостела Даниэль снизилась на 18% к аналогичной неделе прошлого года при росте коттеджей на 5%
> — Food cost ресторана вырос до 42% выручки (исторический коридор нормы — 22–27%)
> — Списания "без списания" составили 87 000 ₽ за неделю — рост в 2,1 раза к прошлой неделе
> — Доля неотвеченных звонков — 27% (норма <10%)

#### Раздел Б. Возможные причины
ИИ анализирует каждое отклонение и предлагает наиболее вероятные объяснения с учётом сезонности, исторических паттернов и отраслевого контекста.

#### Раздел В. Рекомендации
Конкретные, применимые рекомендации по каждой проблемной точке — сформулированные под конкретные цифры и контекст объекта.

---

### Лист 7 — Сырые данные (служебный)

Технический лист: сюда записываются данные, полученные от API Travelline и IIKO, до обработки. Не предназначен для просмотра пользователями — используется как источник для остальных листов.

## 5. Логика выявления аномалий

Система **сама определяет пороги** на основе исторических данных.

Алгоритм:
1. Для каждого показателя рассчитывается среднее значение за аналогичные недели прошлых лет (с учётом сезонности)
2. Отклонение более чем на 1,3 от исторической нормы → аномалия
3. Аномалии ранжируются по значимости (влияние на выручку и/или прибыль)
4. В ИИ-промпт передаются: список аномалий, их размер, сезонный контекст, данные операционного журнала

**Пороги-сигналы (предварительные, подлежат калибровке):**

| Показатель | Люфт нормы | Сигнал предупреждения | Критический сигнал |
|---|---|---|---|
| Загрузка | ±10% к прошлому году | >10% | >20% |
| ADR (Средняя цена) | ±10% | >10% | >20% |
| Food cost | исторический коридор | +3 пп | +5 пп |
| Отмены бронирований | ±5% | >5 пп | >10 пп |
| Списания "без списания" | ±20% к прошлой неделе | >20% | >50% |
| % неотвеченных звонков | <10% | 10–20% | >20% |
| Доля прямых бронирований | изменение >5 пп | — | — |

> **Примечание:** На этапе запуска предоставляется документ с предварительными пороговыми значениями, который анализируется на корректность и при необходимости корректируется.

## 8. Что будет в системе

- Автоматический сбор данных из Travelline и IIKO без ручного ввода
- Раздельная аналитика по сегментам: коттеджи / хостел Ален / хостел Даниэль
- Детальная аналитика ресторана: по времени суток, кухня vs бар, food cost, списания
- Сравнение с аналогичным периодом прошлого года (YoY) и предыдущей неделей (WoW)
- Автоматическое обнаружение аномалий на основе исторических данных
- Операционный журнал с ручным вводом нечисловых данных
- ИИ-резюме с причинами отклонений и конкретными рекомендациями
- Визуальный дашборд с цветовыми индикаторами и спарклайнами
- Двойная доставка: Telegram + email каждый понедельник

---

## 9. Чего НЕ будет в системе

- Полной автоматизации всех данных (часть требует ручного ввода: инциденты, звонки, уборки, погода)
- Прогнозирования и бюджетирования (не входит в текущий объём)
- Управления бронированиями или заказами (только чтение данных)
- Модуля HR / расчёта зарплат
- Мобильного приложения (только Google Таблица)
- Разграничения прав доступа между пользователями
- Интеграции с системами, кроме Travelline и IIKO (кроме возможного чтения из других Google Таблиц с нужными данными)
- Финансовой отчётности (P&L, баланс) — только операционные показатели
