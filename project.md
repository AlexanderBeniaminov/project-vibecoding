# Проект: Автоматический еженедельный отчёт — Курорт Губаха

## 1. Контекст и цель

**Объект:** Всесезонный курорт Губаха — единый объект, включающий:
- 33 коттеджа
- 2 здания хостела (~200 гостей одновременно)
- Ресторан на 90 посадочных мест

**Проблема, которую решает проект:**
- Еженедельные отчёты заполняются вручную — это занимает время и создаёт риск ошибок и опечаток
- Нет централизованного взгляда на операционную деятельность отеля и ресторана в одном месте
- Нет автоматического выявления проблемных точек и рекомендаций по их устранению

**Цель:** Создать автоматизированную систему, которая каждый понедельник утром самостоятельно собирает данные за прошедшую неделю из Travelline и IIKO, формирует отчёты в Google Таблице и генерирует ИИ-анализ отклонений с рекомендациями.

---

## 2. Для кого

| Пользователь | Роль |
|---|---|
| Составитель отчёта (вы) | Аналитик / операционный менеджер |
| Руководитель проекта | Управляет отелем и рестораном |

Разграничение доступа не требуется — оба пользователя видят всё.

---

## 3. Источники данных

| Система | Тип доступа | Данные |
|---|---|---|
| **Travelline** | API | Бронирования, загрузка, выручка по сегментам, отмены |
| **IIKO** | API | Выручка ресторана, гости, чеки, себестоимость, списания |

Исторические данные в обеих системах доступны — используются для сравнения и обнаружения аномалий.

---

## 4. Структура Google Таблицы

Таблица состоит из **5 листов:**

---

### Лист 1 — Отель: Текущая неделя

Данные за отчётную неделю (пн–вс) по двум сегментам раздельно: **Коттеджи** и **Хостел**, и суммарно.

**Показатели:**

| Показатель | Аббревиатура | Расшифровка |
|---|---|---|
| Выручка | — | Общая и по сегментам (коттеджи / хостел) |
| Загрузка | Occupancy (Загрузка) | % занятых номеров/коттеджей от доступных |
| Средняя цена номера | ADR (Средняя цена) | Средняя дневная ставка |
| Доход на доступный номер | RevPAR (Доход на номер) | Выручка / всего доступных номеро-ночей |
| Количество гостей | — | Въезды, ночёвки, выезды |
| Отмены и no-show | — | Количество и % от бронирований |
| Источники бронирований | — | OTA, прямые, корпоративные и др. |
| Средняя длина пребывания | LOS (Срок пребывания) | Среднее количество ночей |

---

### Лист 2 — Отель: Сравнение периодов

Для каждого показателя — три колонки:

| Колонка | Описание |
|---|---|
| Текущая неделя | Фактические данные |
| Аналогичная неделя прошлого года | Данные WoW (неделя к неделе, год к году) |
| Отклонение | Абсолютное и % |

Цветовое выделение: рост — зелёный, снижение — красный, нейтрально — серый.

---

### Лист 3 — Ресторан: Текущая неделя + сравнение

**Показатели:**

| Показатель | Описание |
|---|---|
| Выручка за неделю | Общая, по сменам (завтрак / обед / ужин) |
| Количество гостей | Всего за неделю, среднее в день |
| Средний чек | На гостя, на стол |
| Себестоимость | Сумма и % от выручки (food cost) |
| Потери и списания | По позициям, итого |
| Топ-10 блюд | По выручке и по количеству |

Каждый показатель — с колонками: текущая неделя / аналогичная неделя прошлого года / отклонение.

---

### Лист 4 — Сводный дашборд

Один экран с ключевыми цифрами по отелю и ресторану за неделю. Визуальные элементы: спарклайны, цветовые индикаторы (светофор), мини-графики динамики за последние 8 недель.

Назначение: быстрый взгляд на состояние объекта за 30 секунд.

---

### Лист 5 — ИИ-анализ: Проблемные точки и рекомендации

Автоматически генерируется через API языковой модели (Claude или GPT-4).

**Структура листа:**

#### Раздел А. Выявленные отклонения
Список аномалий, определённых системой на основе сравнения с историческими данными. Пример:

> — Загрузка хостела снизилась на 18% относительно аналогичной недели прошлого года при росте коттеджей на 5%
> — Себестоимость ресторана выросла до 42% выручки (норма по историческим данным — 32–35%)
> — Количество отмен выросло в 2,3 раза по сравнению со средним за 8 недель

#### Раздел Б. Возможные причины
ИИ анализирует каждое отклонение и предлагает наиболее вероятные объяснения с учётом сезонности, исторических паттернов и отраслевого контекста.

#### Раздел В. Рекомендации
Конкретные, применимые рекомендации по каждой проблемной точке. Не шаблонные — сформулированные под конкретные цифры и контекст объекта.

---

## 5. Логика выявления аномалий

Система **сама определяет пороги** на основе исторических данных — не требует жёстко заданных KPI.

Алгоритм:
1. Для каждого показателя рассчитывается среднее и стандартное отклонение за скользящее окно (8–12 недель + аналогичные недели прошлых лет)
2. Отклонение более чем на 1,5–2σ от нормы → аномалия
3. Аномалии ранжируются по значимости (влияние на выручку)
4. В ИИ-промпт передаются: список аномалий, их размер, сезонный контекст, исторические данные

> **Примечание:** На этапе запуска совместно определяются базовые плановые показатели для первичной калибровки системы.

## 6. Расписание и доставка

| Параметр | Значение |
|---|---|
| Период отчёта | Пн–Вс (календарная неделя) |
| Время генерации | Воскресенье ночью / понедельник ранним утром (автозапуск) |
| Формат доставки | Ссылка на Google Таблицу (всегда актуальна) + уведомление на email или в мессенджер |
| Язык | Русский. Профессиональные аббревиатуры дублируются: ADR (Средняя цена), RevPAR (Доход на номер), LOS (Срок пребывания), Occupancy (Загрузка) |

## 7. Технический стек (план реализации)

| Компонент | Инструмент |
|---|---|
| Источник данных (отель) | Travelline API |
| Источник данных (ресторан) | IIKO API |
| Хранение и отображение | Google Sheets (Google Таблицы) |
| Автоматизация | Google Apps Script (триггер по расписанию) |
| ИИ-анализ | Claude API или OpenAI GPT-4 API |
| Уведомления | Email (Gmail) или Telegram Bot |

Реализация: самостоятельно (vibe coding).

---

## 8. Что будет в системе

- Автоматический сбор данных без ручного ввода
- Сравнение с аналогичным периодом прошлого года (YoY) и предыдущей неделей (WoW)
- Раздельная аналитика по сегментам: коттеджи vs хостел
- Автоматическое обнаружение аномалий на основе исторических данных
- ИИ-резюме с причинами отклонений и конкретными рекомендациями
- Визуальный дашборд с цветовыми индикаторами
- Отчёты на русском языке с расшифровкой терминов

---

## 9. Чего НЕ будет в системе

- Ручного ввода данных менеджером
- Прогнозирования и бюджетирования (не входит в текущий объём)
- Управления бронированиями или заказами (только чтение данных)
- Модуля HR / персонала
- Мобильного приложения (только Google Таблица)
- Разграничения прав доступа между пользователями
- Интеграции с другими системами, кроме Travelline и IIKO
- Финансовой отчётности (P&L, баланс) — только операционные показатели
